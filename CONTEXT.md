# CONTEXT

## Purpose

A 3D viewer of underground infrastructure in Singapore, intended to help urban planners identify safe zones for future underground construction and surface potential clashes with existing utilities.

The tool is built on **public/open data only**. Real underground utility data (water, sewer, power, telecoms) is held by individual asset owners (PUB, SP Group, telcos) and is not publicly accessible. The tool therefore operates as a **demonstration-grade clash-detection viewer**: it shows what such a tool would look and feel like, using public data plus procedurally-inferred utility corridors, with a clean data-ingestion seam designed to accept real agency feeds in future.

## Glossary

- **Clash** — A spatial conflict between a proposed underground work and an existing underground asset, defined as the proposed work's volume being within a **clearance buffer** of the asset's volume (not strict geometric intersection). The buffer is a single global distance in v1, configurable by the planner; per-asset-class buffers graded by criticality are a designed-but-deferred upgrade.
- **Clearance buffer** — The distance within which a proposed work is considered to clash with an existing asset. Reflects real-world safety standards for working near live utilities, not just geometric overlap.
- **Clash confidence** — Every clash result carries the confidence of the asset it hit: **real** (against authoritative geometry such as an MRT tunnel) or **inferred** (against a utility bundle). Real and inferred clashes are surfaced in the UI distinctly, so the tool is never dishonest about what it actually knows.
- **Clash detection (hybrid)** — Runs in two modes. **Live coarse feedback** while the planner is sketching a proposed work: nearby assets light up as the sketch moves, using a fast bounding-volume test. **Full clash report** when the planner commits the sketch: precise distances, asset metadata, and confidence breakdown for every clash and near-miss. The live mode prioritizes interactivity; the committed mode prioritizes correctness and presentability. In v1 both modes run in a browser **Web Worker** off the main thread; in v2 the committed-mode computation moves to a backend service alongside accounts and collaboration. The clash interface is async request/response so this migration is a swap, not a rewrite.
- **Clash report** — The committed-mode output. Surfaced two ways: (a) clashing assets highlighted in the 3D viewer, and (b) a sidebar list with one row per clash. Each row carries proposed-work name, asset name and class, **plan distance** and **vertical distance** as separate values (engineers think in plan vs. depth, not in 3D Euclidean), and the asset's confidence tier. Clicking a row flies the camera to the clash. The whole report is exportable as a **Markdown** document with a header (study-area bbox, buffer setting, timestamp, tool version) so it can be pasted into emails, planning submissions, or version-controlled. PDF export is deferred — Markdown covers v1's sharing needs without a generation toolchain.
- **Safe zone** — A subsurface volume with no known or inferred underground assets, suitable for new construction.
- **Asset** — Any underground feature represented in the viewer (tunnel, pipe, cable, duct).
- **Real asset** — An asset whose geometry comes from authoritative agency data. (Applies fully only to future ingested feeds — see Confidence tiers below.)
- **Confidence tier** — Every asset carries one of three tiers, reflecting how trustworthy its geometry is. The clash report and visual styling both surface the tier so the tool never overclaims:
  - **`real`** — Plan and depth both from authoritative source. Reserved for future ingested agency data.
  - **`real-plan`** — Plan from a public authoritative source (LTA DataMall, OSM, PUB publications); depth assumed from a documented class default (e.g. shallow MRT line ~15m, deep MRT line ~30m, road tunnel ~10m, pedestrian underpass ~5m, DTSS deep). Every asset of this tier records its `depthSource: "assumed-class-default"` so a future real-depth value swaps in surgically per-asset.
  - **`inferred`** — Geometry procedurally generated from proxies (currently only the utility bundle).
- **Inferred asset** — An asset whose geometry is procedurally generated from public proxies (e.g. a utility corridor inferred to run under a road centerline at standard depth). Always visually distinguished from real assets.
- **Utility bundle** — A single inferred asset representing the typical bundle of small utilities (water, sewer, power, telecom) that runs beneath a road. Rendered as one visually-banded volume rather than four separate lines, because at demo confidence we do not pretend to know which specific cable is at which depth. Splits into individual classes when real agency data is ingested. Generated by a **build-time bake** from OpenStreetMap Singapore road centerlines (fetched via Overpass API, cached in the repo for reproducible offline builds) and stored as **tiles** (~1 km² each, ~735 tiles covering Singapore). The viewer fetches only the tiles intersecting the current study area.
- **Tile** — A bounded geographic cell (~1 km²) used as the unit of asset pre-baking and on-demand loading. Tile size is a tuning parameter; the goal is bounded per-session download.

## Coordinate system

- **Storage & exchange**: SVY21 (EPSG:3414, Singapore Transverse Mercator) for horizontal coordinates; depths in metres below ground level (**mBGL**). This is what Singapore agencies, planners, and any future real data feed actually use.
- **In-memory app model**: same as storage — SVY21 + mBGL. All distance calculations, clash maths, buffer offsets, and reasoning happen in a flat metric CRS where 1 unit = 1 metre.
- **Renderer**: WGS84 (Cesium's native frame). Conversion happens at the edge, just before geometry is handed to Cesium, using `proj4js`. Cesium is a rendering sink, not a source of truth.
- **Depth datum**: mBGL throughout. Conversion to ellipsoidal height (for Cesium) happens at the same edge as the horizontal conversion, using terrain height as the ground reference.

See [ADR 0001](docs/adr/0001-svy21-mbgl-as-internal-crs.md).

## Tech stack

- **Rendering**: CesiumJS (3D viewer, terrain, 3D Tiles, subsurface-native).
- **Basemap**: OneMap raster tiles (Grey style) as the imagery layer; Cesium World Terrain for elevation; extruded OSM building footprints as a toggleable above-ground reference layer.
- **App shell**: React + Vite. Cesium is mounted directly inside React (a `resium` bridge is acceptable but not required). No SSR, no Next.js — the app is a 100% client-side SPA deployable as static files.
- **Compute**: Clash detection runs in a browser Web Worker in v1. The interface is async request/response so the v2 migration to a backend service is a swap.
- **Persistence**: URL-encoded study-area state in v1 (no backend, no accounts). The state shape is designed for migration to account-backed persistence in v2.

## Project structure & deployment

- **Repo shape**: Single repo, single package. `src/` for the React app, `scripts/` for the build-time bake pipeline, `data/` for baked tiles and source caches (Overpass exports, LTA DataMall snapshots).
- **Baked data**: Committed via **Git LFS** so the repo stays performant while baked tiles remain versioned and reproducible. Rollbacks work; the bake is reproducible from the source caches in `data/sources/`.
- **Deployment**: Cloudflare Pages, default `*.pages.dev` URL (no custom domain in v1). Chosen over GitHub Pages because Cloudflare's edge functions give a no-friction upgrade path when v2 introduces a small backend for accounts.

## Testing strategy

The bugs that matter in this tool are **geometric**, not interactional. A wrong distance computation produces a silently wrong clash. Testing is weighted accordingly:

- **Heavy unit tests** (Vitest) on the load-bearing pure code: coordinate conversion (SVY21 ↔ WGS84, mBGL ↔ ellipsoidal), clash maths (distance, buffer test, plan-vs-vertical decomposition), the bake pipeline (OSM road segment → utility bundle geometry → tile), tile-selection logic, and the clash-worker message protocol.
- **One Playwright smoke test** for the golden path: open the app at a fixture URL, draw a box, see clashes, export the report. Verifies the seams work end-to-end.
- **No React component tests** for panels unless one earns it by getting genuinely complex.
- **A committed fixture study area** (a ~1 km² slice with hand-curated MRT / road / utility data) is the substrate for both bake-pipeline tests and the E2E. Reproducible and version-controlled — no live dependency on Overpass or LTA DataMall during test runs.

## Asset classes (initial set)

| Class | Source confidence |
|---|---|
| MRT tunnels | Real (LTA / OneMap) |
| Road tunnels | Real (LTA / OneMap) |
| Pedestrian underpasses | Real (OSM) |
| Deep Tunnel Sewer System (DTSS) | Real-schematic (PUB public alignments) |
| Utility bundle | Inferred (road centerline + standard depth band) |

Common Services Tunnels (CST), building basements, and individual utility lines are out of scope for the initial set. CST geometry from public planning docs is too schematic to surface without undermining the tool's credibility; the others enter the model when real data is ingested.
- **Asset owner** — The agency or company that owns an asset class (e.g. PUB owns water/sewer, SP Group owns power).
- **Ingestion seam** — The interface through which real agency data could be loaded into the viewer in future, replacing or augmenting inferred assets.
- **Viewer** — The CesiumJS-based 3D scene where assets, terrain, and basemap are rendered. The viewer is the application's primary surface; all interaction (selection, clash highlighting, navigation) happens here or in panels attached to it.
- **Layer** — A toggleable category of content in the viewer. Examples: basemap (OneMap Grey), terrain, buildings (extruded OSM footprints), MRT tunnels, inferred utility corridors, proposed works. The user can show/hide each layer independently to switch between above-ground and subsurface contexts.
- **Visual encoding** — A two-axis convention used everywhere in the viewer and the clash report:
  - **Color = asset class** (deep blue: MRT; dark grey: road tunnel; tan: pedestrian underpass; brown: DTSS; yellow-amber: utility bundle).
  - **Edge / opacity = confidence tier** (solid mesh with slight transparency: `real-plan`; semi-transparent and dashed/striped: `inferred`).
  - **Proposed works** sit in their own distinct visual register — bright cyan with a glowing outline — so they never blend with existing assets regardless of class or confidence.
  - **Clash highlight** is red, applied on top of the asset's normal styling. The red itself inherits the asset's confidence treatment: **solid red outline** for clashes against `real-plan` assets, **dashed red outline** for clashes against `inferred` assets. The sidebar row also carries a confidence chip so the distinction reads in the list and in the 3D scene.
- **Proposed work** — A planner's sketch of a future underground construction, used as the subject of clash detection. Two shapes are supported: a **box** (footprint polygon + top/bottom depth, for excavations and basements) and a **corridor** (polyline + diameter + depth profile, for pipes, cables, and tunnels). A study area may contain multiple proposed works simultaneously so the planner can compare options; each has its own identity, name, visibility toggle, and clash report. File upload (DXF/IFC/glTF) is a designed-but-deferred ingestion path that produces the same internal box/corridor representation.
- **Study area** — A planner-selected subregion of Singapore (typically 1–5 km²) loaded in full 3D detail. The unit of analysis for clash detection. The whole-island view is context only; detailed work always happens inside a study area. A study area is defined by a rectangle on the map and is **persistent** — encoded in the URL so it survives refreshes and can be shared with colleagues. URL-only persistence today; the state shape is designed to migrate to account-backed persistence (saved named study areas, annotations, collaboration) later without rework.
