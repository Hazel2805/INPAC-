# 0002 — Baked utility-bundle tiles are ODbL

**Status**: Accepted

## Context

The tool's inferred *utility bundle* asset is generated by a build-time bake script. The bake takes road centerlines from OpenStreetMap Singapore as input and emits 3D corridor geometry under each road, sliced into ~1 km² tiles that the viewer fetches on demand.

The application's source code will be released under MIT. The natural assumption — and the assumption a reader will start with — is that everything in the repo, including these baked tiles, is therefore MIT.

This is wrong, and silently so.

OpenStreetMap data is licensed under the **Open Database License (ODbL) 1.0**. ODbL is a *share-alike* licence for derivative databases. The baked tiles are a derivative database: their geometry is computed directly from OSM road centerlines (offset and extruded), and their value is meaningfully tied to OSM's content. Under ODbL §4.4, a publicly distributed derived database must itself be offered under ODbL.

If we ship the baked tiles inside the app deployment (which we will — they are required content) and label them MIT along with the rest of the repo, we are misrepresenting their licence and exposing the project to a clear-cut ODbL violation. The remedy is straightforward and costs nothing — but it has to be deliberate, because the default assumption ("one repo, one licence") would silently produce the wrong outcome.

## Decision

- The repository as a whole uses a **mixed licence**, documented in `LICENSE` and `ATTRIBUTION.md`.
- **Source code, build scripts, configuration, and documentation**: MIT.
- **Baked tiles in `data/tiles/` (any geometry derived from OSM): **ODbL 1.0**, with attribution to "OpenStreetMap contributors" in every tile file's metadata header and in the viewer chrome.
- The bake pipeline writes the ODbL notice and OSM attribution into each emitted tile as part of the build, so the licence travels with the data.

## Consequences

**Positive**
- Compliant with OSM's licence terms.
- A future contributor who reuses the baked tiles in another project inherits the obligation correctly because it's on the tile, not just in the repo root.
- No legal review required — this is the conventional posture for OSM derivatives.

**Negative**
- The repo licensing is no longer a single sentence. New contributors must read `ATTRIBUTION.md`.
- A future move away from OSM as the road source (e.g. switching to OneMap roads if SLA opens them) would let us re-licence the tiles — but until then, the constraint binds.

## Alternatives considered

- **MIT everything.** Rejected: an ODbL violation.
- **ODbL the whole repo.** Rejected: ODbL is not a software licence; applying it to source code is meaningless and confuses contributors.
- **Don't ship the tiles; recompute on the user's machine at first load.** Rejected: would still create an ODbL derivative database in the user's browser cache, and would multiply load time by the bake cost on every cold visit.
